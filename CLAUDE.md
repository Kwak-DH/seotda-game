# 섯다 변형 규칙 엔진 + React UI

`packages/core` (프레임워크 독립적 순수 TypeScript 규칙 엔진)와 `packages/ui` (React 클라이언트) 워크스페이스로 구성된 섯다 변형 게임 프로젝트.

## 정확한 섯다 규칙

### 카드 (20장 확정)

**1~10월, 월별로 서로 다른 두 종류(variant)씩 총 20장.** 11월·12월 패는 없다.

> 최초 구현 시 "10장(1~10월 각 1장)" 체계로 잘못 설계되었던 적이 있다. **20장이 맞는 스펙**이므로 혼동하지 말 것 — 자세한 경위는 아래 "지금까지 발견하고 고친 버그" 참고.

| 월 | variant A | variant B |
|---|---|---|
| 1월 | 광 (송학+태양) | 홍단 (송학+붉은 띠) |
| 2월 | 열끗 (매화+꾀꼬리) | 홍단 (매화+붉은 띠) |
| 3월 | 광 (벚꽃+휘장) | 홍단 (벚꽃+붉은 띠) |
| 4월 | 열끗 (등나무+두견새) | 초단 (흑싸리+띠) |
| 5월 | 열끗 (창포+다리) | 초단 (난초+띠) |
| 6월 | 열끗 (모란+나비) | 청단 (모란+파란띠) |
| 7월 | 열끗 (싸리+멧돼지) | 홑껍데기 (싸리+띠) |
| 8월 | 광 (보름달+기러기) | 열끗 (억새+기러기) |
| 9월 | 열끗 (국화+술잔) | 청단 (국화+파란띠) |
| 10월 | 열끗 (단풍+사슴) | 청단 (단풍+파란띠) |

족보 계산(땡/끗수/알리~세륙/광땡)은 **월 숫자만으로** 판정하며 variant와 무관하다. variant 구분이 필요한 건 아래 특수패뿐이다.

### 광땡

광패는 1월·3월·8월 세 장뿐. `38광땡(3+8) > 18광땡(1+8) > 13광땡(1+3)`.

### 특수패 (variant 조합에 의존)

**47 암행어사** — 정확히 4월 열끗[두견새] + 7월 열끗[멧돼지] 조합.
- 평상시엔 그냥 1끗 패(4+7=11)로 취급된다.
- 상대가 13광땡 또는 18광땡일 때만 그 상대를 이긴다.
- **38광땡은 무적** — 암행어사도 못 이긴다.

**37 땡잡이** — 정확히 3월 광 + 7월 열끗[멧돼지] 조합.
- 평상시엔 망통(3+7=10→0끗)으로 취급된다.
- 상대가 1땡~9땡일 때만 그 상대를 이긴다.
- **10땡(장땡)은 못 이긴다.**

**94 구사 / 멍텅구리구사** — 4월+9월 조합.
- 일반 구사: 4월 아무 variant + 9월 아무 variant. 상대가 알리 이하 족보(땡·광땡을 제외한 전부: 알리/독사/구삥/장삥/장사/세륙/일반 끗수/망통)면 무효 처리 + 재경기.
- 멍텅구리구사: 정확히 4월 열끗[두견새] + 9월 열끗[술잔]일 때만. 상대가 9땡 이하(모든 땡 포함)면 무효 처리 + 재경기. 단 광땡·암행어사·땡잡이·10땡(장땡)은 명시적으로 제외(무효화되지 않음).
- 멍텅구리구사 조합은 일반 구사보다 먼저 판정해 더 강한 특수패로 우선 인식한다.

### 재경기(구사 무효화) 시스템

- 구사/멍텅구리구사가 발동하면 `GameState.phase`가 `"rematch"`로 전환된다.
- **팟은 리셋되지 않고 그대로 이월**된다.
- 무효화된 판에서 다이했던 플레이어는 `excludedFromRematch`로 표시되어 재경기에서 제외된다(카드를 받지 않고 다이 상태로 대기).
- 제외된 플레이어는 `rejoinRematch(playerId)`로 재참여 배팅을 해야 다음 판에 다시 낄 수 있다. 배팅 비율은 `GameState.rebuyRatio`로 **설정 가능**하며 기본값은 **0.5(팟의 50%)**다.
- 암행어사/땡잡이로 승부가 갈리면 `GameState.specialWin: { kind, winnerId } | null`에 어떤 특수패로 누가 이겼는지 기록된다(일반 승부/재경기 시엔 `null`).

## 지금까지 발견하고 고친 버그

- **카드 체계를 "10장(1~10월 각 1장)"에서 "20장(월별 2종류 variant)"으로 재정정.** 최초 커밋은 10장 체계로 잘못 구현되어 있었는데, 사용자가 나무위키 자료와 직접 그린 카드 이미지를 근거로 제공해 정확한 20장 스펙을 확정했다.
- UI(`useGameEngine`/`Table.tsx`)가 core의 재경기 API(`phase === "rematch"`, `rejoinRematch`)를 전혀 반영하지 못해, 구사가 발동하면 베팅 패널이 비활성화된 채 화면이 멈추고 빠져나갈 방법이 없었다.
- `useGameEngine`의 `startNewRound`가 매번 `createInitialGameState`로 완전히 새 `GameEngine`/`GameState`를 만들어, 재경기 상황에서도 `phase`가 `"waiting"`으로 초기화되고 팟이 0으로 리셋되어 구사 이월이 깨졌다. 기존 엔진 인스턴스를 그대로 이어받아 `engine.startNewRound()`를 호출하도록 수정.
- `currentPlayerIndex`가 판이 끝나도 리셋되지 않아, 직전 판이 홀수 번의 베팅 액션(예: 콜→AI 레이즈→콜)으로 끝나면 다음 판이 AI 차례로 시작된 것처럼 되어 사람 플레이어 쪽 베팅 패널이 영구히 멈췄다. `dealCards`/`dealRematchCards`에서 살아있는 첫 플레이어로 되돌리도록 수정.
- `applyBetAction`의 "call" 분기가 이번 라운드에 이미 베팅한 금액(`totalBet`)을 고려하지 않고 직전 베팅액(`lastBetAmount`) 전체를 매번 또 더해, 레이즈에 대응해 두 번째로 콜할 때 금액이 이중 계산되어 베팅 총액이 영원히 맞지 않는(=라운드가 끝나지 않는) 버그가 있었다. 부족분(`lastBetAmount - totalBet`)만 채우도록 수정.

## 현재 구현 상태

- `packages/core`: 카드/족보/특수패/베팅/재경기/기본 AI 규칙 엔진. 테스트 **95개** 전부 통과, `tsc --noEmit` 클린.
- `packages/ui`: React 클라이언트. 구사 재경기 배너, 특수패 승리 안내, 재참여 버튼까지 core와 연동 완료. `tsc -b` 클린.
- 특수패(암행어사/땡잡이/구사/멍텅구리구사)와 재경기 시스템 구현 완료, 실제 브라우저 플레이로 종단 검증됨.
